## Issues to fix

### 1. Logo is requested at 1920px width on every page (HIGH)

Every page has `<img src="/_next/image?url=/images/logo-white.png&w=1920&q=75">` for the site-wide logo. A header logo rendering at ~160px wide shouldn't be fetching a 1920px source.

**Cause:** The Image component that wraps the logo is missing a `sizes` prop, so Next.js defaults to the largest breakpoint (1920px).

**Fix:** In whichever component renders the header logo (likely `src/components/layout/Header.tsx`), add `sizes="160px"` (or the actual display width) to the `Image`.

**Impact:** Saves ~80–200KB per page-load on every visit. Logo ships on header AND footer on every page, so this is the single biggest byte-saving win.

### 2. No `priority` hint on hero images (MODERATE)

On `/blog/zone-2-training-complete-guide`, the hero image (`/api/og/blog-hero?slug=...`) renders as `<img>` with no `loading` attribute and no `fetchpriority="high"`.

Without `fetchpriority="high"`, the browser discovers the hero after HTML parse completes AND after other preloaded resources — delaying LCP by several hundred ms on slow connections.

**Fix:** In `src/app/blog/[slug]/page.tsx` (and similar for podcast hero), set `priority={true}` on the hero `<Image>`. Next.js translates this to `fetchpriority="high"` + `loading="eager"`.

**Impact:** Expected LCP improvement of 200–600ms on mobile.

### 3. `/coaching`, `/tools/ftp-zones`, `/plan/wicklow-200` have no hero `<img>` (verify)

Only the logo appears as `<img>` on these pages. Either they use CSS background-image for the hero (doesn't block text rendering, good) or they genuinely have no hero image (which for /coaching is a conversion/LCP question worth checking).

**Action:** Manually verify in the browser that /coaching renders what you expect above the fold. If the hero is CSS-based, that's fine. If there's supposed to be a hero image and it's not rendering, that's a separate bug.

## What I did NOT measure (needs a real Lighthouse run)

I could not run Lighthouse from the sandbox — Chrome headless isn't available, and `pagespeedonline.googleapis.com` is not on the egress allowlist. These metrics require a Lighthouse run on your machine:

- **LCP** (Largest Contentful Paint) — actual paint timing with throttling
- **CLS** (Cumulative Layout Shift) — only measurable in a running browser
- **INP** (Interaction to Next Paint) — requires real interaction
- **TBT / TTI** (main-thread blocking) — requires throttled CPU run

### Fastest way to get these numbers

Open Chrome DevTools → Lighthouse tab → run on each of the 4 URLs with "Mobile" and "Simulated throttling". Save the JSON reports for before/after comparison once the two fixes above ship.

Alternatively, enter each URL at <https://pagespeed.web.dev/> — the Google-hosted version, no auth needed, gives both lab (Lighthouse) and field (CrUX) data if the site has enough traffic.
